# Windows installer automation for TLS-key-log + packet capture (Option C)

**Research date:** 2026-07-20  
**Scope:** A Windows companion for THE FINALS that captures the local player's HTTPS traffic using TLS session-key logging (`SSLKEYLOGFILE`) and Npcap/tshark. This is an installer and operating-system feasibility assessment, not permission to capture the traffic. The contractual and enforcement risks remain material: the Nexon terms explicitly prohibit packet sniffing and intercepting/reading service data; see [the companion ToS research](./the-finals-tls-capture-tos-enforcement.md).

## Verdict

An installer can automate most preparation **after the player accepts elevation and any required third-party licence terms**:

- install the companion and its capture executable;
- install Npcap silently **only under an Npcap OEM Redistribution licence**;
- create a protected key-log directory, configure the capture filter, discover and persist the preferred active network interface, and start/stop `tshark`;
- persist `SSLKEYLOGFILE` for future processes without rebooting Windows; and
- detect Npcap's driver/service after installation and prompt for a reboot only when its installer returns the documented reboot-required result.

It cannot truthfully promise a zero-click or zero-restart path:

1. **Npcap adds a signed kernel/NDIS capture driver.** The install needs an elevated installer/UAC approval. The free Npcap package does not grant a product vendor unlimited redistribution or silent installation; a commercial companion needs Npcap OEM Redistribution rights. A silent child installer does not eliminate the bootstrapper's Windows UAC consent.
2. **Environment variables are inherited at process creation, not injected into existing processes.** A system/per-user `SSLKEYLOGFILE` setting affects only processes started after the change and only if their runtime actually supports/consults it. The user must fully exit and restart Steam and THE FINALS (or launch the game through a verified companion-owned launch path). An OS reboot is not normally required for this; it is a broad but reliable way to restart the relevant process tree.
3. **THE FINALS support for `SSLKEYLOGFILE` is an implementation assumption, not a documented Embark guarantee.** The existing extractor demonstrates that its target build has used this route, but it is not proof for every build, renderer, launcher, or future anti-cheat policy. Do not claim that “Chrome/Chromium/CEF/Steam games” universally honour the variable.
4. A first run still needs an informed **ToS/account-risk acknowledgement** and, in normal consumer Windows use, UAC approval. Given the terms risk, this acknowledgement is product-safety consent, not merely an installer licence checkbox.

### Reboot versus restart: the precise answer

- **Writing an environment variable:** no Windows reboot is required. It does not update a process already running.
- **Steam and THE FINALS:** fully exit them and relaunch them after the variable is set. Merely closing a game window may leave launcher/helper processes alive; verify them as exited or offer a “Restart Steam” instruction.
- **Npcap:** no reboot on the normal successful path, but the installer may return **3010**, which Npcap defines as “success, reboot required before ready to use.” Treat that result as mandatory and defer the capture diagnostic until after reboot.
- **The game:** must start after key logging is in its environment. If a key-log diagnostic fails, do not ask for a Windows reboot as a generic fix; first verify the key-log path, Steam/game restart, and current game compatibility.

## Important boundary: `SSLKEYLOGFILE` is not a universal Windows or Chromium switch

Windows stores an environment string; it does not make applications enable TLS key logging. Each process receives an environment block when it starts, and an application/runtime must explicitly implement the variable. Chromium's source has an `SSLKeyLoggerImpl` that opens a supplied path and appends generated key-log lines, but the cited implementation does not say that every Chromium embedder reads `SSLKEYLOGFILE`. OpenSSL likewise exposes a key-log callback API for applications; the application decides whether to set it. [OpenSSL `SSL_CTX_set_keylog_callback`](https://docs.openssl.org/3.4/man3/SSL_CTX_set_keylog_callback/)

Therefore:

- **Chrome / Chromium / CEF:** do not generalize from their lineage. Test the exact executable and version; Chrome may use an explicit key-log configuration/switch in a particular release, while an embedded CEF client can omit it entirely.
- **Steam:** Steam itself is not a TLS-library compatibility layer for games. Its only relevant role here is whether it is the parent/launcher of a game process and therefore whether it propagates an environment it already has.
- **THE FINALS:** current support must be a release-gated diagnostic. The baseline extractor's setup asks for system `SSLKEYLOGFILE`, reboot, Wireshark, and manual interface selection, which demonstrates a historical workflow—not a supported API or future-proof behaviour.

## Recommended minimum first-run flow

### Installer-owned work

1. Show a pre-elevation screen naming **Npcap**, the capture purpose, its driver/network implications, and the game-account/ToS risk. Link the full risk disclosure and provide a non-capture/manual fallback.
2. Obtain explicit capture-risk acknowledgement and Windows UAC elevation.
3. Install/upgrade Npcap OEM silently, wait for its result, verify the service/DLL, and immediately stop for a reboot if it reports `3010`.
4. Install the companion and its compliant `tshark` distribution (or use a separately installed capture provider); write licence notices and corresponding-source materials where the Wireshark GPL requires them.
5. Create a per-user key-log/capture directory with restrictive ACLs. Configure a per-user environment variable **only if a process-scoped launch is not technically verified**. Never enable it system-wide by default.
6. Run `tshark -D` plus a short non-game diagnostic to preselect the active default-route interface; persist the result and retain a fallback picker.

### Shortest honest player sequence

1. Read and accept the explicit account/ToS, local-traffic, and TLS-key-log risk disclosure; approve UAC (or ask their administrator).
2. If Npcap requested a reboot, reboot once. Otherwise, **fully exit Steam and THE FINALS**, then relaunch them (or use the companion's validated dedicated launch route).
3. In the companion, press **Start capture**. If automatic interface validation cannot see expected target traffic, choose the one adapter from a labelled list (for example Wi-Fi, Ethernet, VPN).
4. Open the required THE FINALS screen / perform the existing extractor's trigger, then let the companion confirm it obtained a key-log line and the expected response. Stop capture and delete/rotate plaintext capture data.

This is realistically **one consent/elevation interaction, one Steam+game restart, and normally zero interface choices**. A reboot is conditional on Npcap's result, not a mandatory step in the normal path. It is still an opt-in advanced setup—not “install and it silently reads your rank.”

## Open risks and validation gates

1. **Licence procurement:** Confirm the current Npcap OEM Redistribution agreement, price, update rights, allowed bundling topology, and whether a separate consent screen is contractually required. The public page establishes that OEM is needed; it is not the full contract.
2. **Wireshark licence compliance:** Inventory the exact `tshark` executable, DLLs, plugins, data files, and licences. Obtain legal review before bundling into a proprietary installer. “No UI” does not make a GPL binary private.
3. **Exact game process tree:** On a clean Windows machine, record which Steam/game/anti-cheat processes survive a normal exit and whether a companion-launched route passes an environment variable to the TLS-owning process.
4. **Exact TLS implementation:** Acceptance-test each supported THE FINALS build: set a temporary process/user key-log destination, start a fresh game session, produce known HTTPS activity, and require a valid NSS key-log line. A zero-line result is an unsupported-build failure, not a reason to keep collecting packets.
5. **Interface automation:** Test Ethernet, Wi-Fi, VPN, Hyper-V/WSL, and multiple adapters. Default-route heuristic plus an opt-in fallback picker is safer than hard-coding a friendly interface name.
6. **Sensitive-data lifecycle:** Threat-model pcap/key retention, crash recovery, support logs, antivirus quarantine, and file permissions. Delete raw capture/key data by default after extracting the minimal result.
7. **ToS/enforcement:** The terms analysis remains a release blocker for any public rollout. Npcap, Wireshark, and Windows feasibility do not make capture permitted by Embark/Nexon.
